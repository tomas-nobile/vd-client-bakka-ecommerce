# Spec 8a: Categorías de Posteos — Taxonomía + Filtro en `/posteos`

> Optimizado para Claude Sonnet.  
> No crear nuevos `.md` de documentación.  
> No ejecutar `npm run build`.

---

## Contexto

La página `/posteos` ya existe con su bloque orquestador en `src/page-posteos/index/`.  
El CPT `social_post` está definido en `src/front-page/includes/social-post.cpt.php` pero **no tiene taxonomía propia**.  
El helper de consulta es `etheme_get_social_posts($count, $offset)` en `src/core/includes/social-posts.helpers.php`.  
El AJAX de "Mostrar más" vive en `src/page-posteos/includes/ajax-handlers.php` con action `etheme_posteos_load_more`.  
El sub-banner admite un parámetro `after_content` (callable) para inyectar contenido debajo — úsalo para el filtro.

Categorías iniciales (deben quedar como términos en WP admin; el spec las registra pero no las inserta con código):
- **Muebles de línea** (slug: `muebles-de-linea`)
- **Muebles a medida** (slug: `muebles-a-medida`)

---

## Cambios requeridos

### 1. Taxonomía: `src/front-page/includes/social-post-category.taxonomy.php` _(archivo nuevo)_

Registrar la taxonomía personalizada `posteo_category` para el CPT `social_post`:

```php
function etheme_register_posteo_category_taxonomy(): void {
    register_taxonomy( 'posteo_category', 'social_post', [
        'labels'            => [
            'name'          => __( 'Categorías de Posteos', 'etheme' ),
            'singular_name' => __( 'Categoría de Posteo', 'etheme' ),
            'all_items'     => __( 'Todas las categorías', 'etheme' ),
            'edit_item'     => __( 'Editar categoría', 'etheme' ),
            'add_new_item'  => __( 'Añadir categoría', 'etheme' ),
        ],
        'hierarchical'      => false,
        'public'            => true,
        'show_in_rest'      => true,
        'rewrite'           => [ 'slug' => 'posteo-categoria' ],
        'show_admin_column' => true,
    ] );
}
add_action( 'init', 'etheme_register_posteo_category_taxonomy' );
```

Luego incluir este archivo en `functions.php`, junto al resto de includes de `front-page`.

---

### 2. Helper de consulta: `src/core/includes/social-posts.helpers.php` _(modificar)_

Agregar el parámetro opcional `$category_slug` a `etheme_get_social_posts()`:

**Firma actual:**
```php
function etheme_get_social_posts( int $count = 15, int $offset = 0 ): array
```

**Firma nueva:**
```php
function etheme_get_social_posts( int $count = 15, int $offset = 0, string $category_slug = '' ): array
```

Cuando `$category_slug` no esté vacío, agregar al `WP_Query` el parámetro:
```php
'tax_query' => [
    [
        'taxonomy' => 'posteo_category',
        'field'    => 'slug',
        'terms'    => sanitize_key( $category_slug ),
    ],
],
```

`etheme_get_home_social_posts()` no cambia (no necesita filtro de categoría en home).

---

### 3. Render de página: `src/page-posteos/index/render.php` _(modificar)_

**a) Leer categoría activa desde query string:**
```php
$active_category = isset( $_GET['categoria'] ) ? sanitize_key( $_GET['categoria'] ) : '';
```

**b) Pasar `$active_category` a `etheme_get_social_posts()`:**
```php
$posts = etheme_get_social_posts( 15, 0, $active_category );
```

**c) Chips de filtro en `after_content` del sub-banner:**

Pasar una callable al argumento `after_content` de `etheme_render_sub_banner()` que renderice:

```html
<div class="posteos-filter" role="tablist" aria-label="Filtrar por categoría">
  <button class="posteos-filter__chip posteos-filter__chip--all [activo si no hay categoría]"
          data-category=""
          role="tab"
          aria-selected="[true|false]">
    Todos
  </button>
  <!-- Por cada término de posteo_category: -->
  <button class="posteos-filter__chip [posteos-filter__chip--active si es la activa]"
          data-category="{slug}"
          role="tab"
          aria-selected="[true|false]">
    {nombre del término}
  </button>
</div>
```

Obtener los términos así:
```php
$terms = get_terms( [ 'taxonomy' => 'posteo_category', 'hide_empty' => false ] );
```

Si no hay términos, no renderizar la barra de filtro.

**d) Pasar el estado de filtro al bloque wrapper** (para que el JS lo lea):
```html
<section ... data-active-category="{$active_category}">
```

**e) Pasar la categoría activa al botón "Mostrar más":**
```html
<button ... data-category="{$active_category}" ...>
```

### 4. AJAX handler: `src/page-posteos/includes/ajax-handlers.php` _(modificar)_

Leer el parámetro `category` del POST y pasarlo al helper:

```php
$category_slug = isset( $_POST['category'] ) ? sanitize_key( $_POST['category'] ) : '';
// ...
$posts = etheme_get_social_posts( $per_page, $offset, $category_slug );
```

### 5. JavaScript: `src/page-posteos/scripts/` _(modificar el módulo de load-more)_

El JS existente maneja el botón "Mostrar más". Agregar:

**a) Chips de filtro:**
- Escuchar `click` en `.posteos-filter__chip`.
- Al hacer clic en un chip:
  1. Marcar ese chip como activo (`posteos-filter__chip--active`, `aria-selected="true"`); desactivar los demás.
  2. Leer su `data-category`.
  3. Vaciar el grid (`#posteos-cards-grid`).
  4. Hacer fetch al endpoint AJAX con `page: 1` y el nuevo `category`.
  5. Insertar el HTML devuelto.
  6. Actualizar el atributo `data-category` del botón "Mostrar más" y resetear su estado de paginación (página siguiente = 2).
  7. Mostrar u ocultar el botón "Mostrar más" según `has_more`.

**b) Botón "Mostrar más":**
- Al hacer clic, leer `data-category` del botón (ya sincronizado por el filtro).
- Enviarlo como `category` en el body del fetch junto al `page` y `nonce` actuales.

**Fetch body de referencia:**
```js
body: new URLSearchParams({
  action:   'etheme_posteos_load_more',
  nonce:    nonce,
  page:     nextPage,
  category: activeCategory,  // '' = todos
})
```

No introducir jQuery ni librerías nuevas.

## Restricciones

- Escapado: `esc_html()` en nombres de términos, `esc_attr()` en slugs, `esc_url()` en URLs.
- Verificar nonce en el handler AJAX antes de procesar (`check_ajax_referer()`).
- No usar jQuery ni librerías JS nuevas.
- No editar `build/`.
- Si `get_terms()` retorna vacío o error, no renderizar la barra de filtro y continuar mostrando todos los posts.
- El filtro de chips es cliente/server híbrido: el chip activo se refleja visualmente vía JS, pero la URL no cambia (sin pushState requerido). Si el usuario recarga, vuelve a "Todos" (comportamiento aceptable para esta iteración).
