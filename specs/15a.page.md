# Spec 15a — «Mi cuenta» logueado: dashboard estilo marketplace

> Complementa **Spec 15** (`specs/15.page.md`). Ámbito: **solo usuario autenticado**. Login, registro y lost-password no deben regresiones.

---

## Prompt ejecutable (Claude Sonnet) — copiar desde aquí

**Role:** Senior WordPress 6.x + WooCommerce + FSE block theme engineer (PHP-first UI, accessible markup).

**Repository / theme:** `bakka` (slug `bakka`). Block namespace `etheme/*`, text domain `etheme`. Workspace: theme root `wp-content/themes/bakka`.

**Prerequisite:** Spec 15 implemented: block `etheme/page-index` under `src/page/index/`, shell `etheme_render_account_shell`, heading `etheme_render_account_heading`, styles `src/page/styles/*.scss`, Contrive tokens (mint bg `#f7fcf4`, accent `#fb704f`, fonts Jost/Archivo, card “no radius” language). **Do not** create a second My Account block; extend this one.

**Objective:** When `is_user_logged_in()`, replace the **default WooCommerce My Account vertical list feel** with a **professional customer dashboard** comparable to large marketplaces (clear information architecture: persistent **side navigation**, optional **dashboard home** with shortcuts and recent activity summaries, **mobile-friendly** nav). Preserve **all Woo default endpoints** (`dashboard`, `orders`, `downloads`, `edit-address`, `edit-account`, `customer-logout`, plugins that add menu items via `woocommerce_account_menu_items`).

**UX / IA (implement, don’t hand-wave):**

1. **Layout (logged-in only):** Two-column pattern from `md` breakpoint up: **left** primary nav (sticky or scroll-safe), **right** main area with current endpoint content. On small screens: **collapsible** nav (button + panel or horizontal scroll with clear labels—pick one pattern and document in code comments only where non-obvious).
2. **Nav items:** Use WC’s real menu structure (filter hooks allowed): labels from WC / translations, correct **active state** for current endpoint, icons optional (inline SVG or simple CSS shapes—**no** icon font CDN). **Logout** visually distinct (destructive-neutral), not lost at the bottom of a wall of links without hierarchy.
3. **Dashboard “home” (`dashboard` endpoint):** Beyond WC’s default welcome text, add a compact **summary strip**: e.g. link to last/open orders, addresses, account details—implemented with **real URLs** from `wc_get_account_endpoint_url()` and counts only if cheap/safe (`wc_get_customer_order_count` or similar; guard for empty).
4. **Visual language:** Stay consistent with Spec 15 (colors, typography, card surfaces). Marketplace polish = **spacing rhythm**, **section titles**, subtle separators—not a new theme skin.
5. **Accessibility:** Focus order logical on mobile when nav opens; `aria-current` on active item; keyboard Esc closes mobile nav if applicable; contrast meets WCAG AA on nav + content.

**Technical constraints:**

- **Rendering:** Prefer new or extended PHP under `src/page/components/*.php` (e.g. `account-dashboard-layout.php`, `account-dashboard-nav.php`) and thin changes in `account-shell.php` **only** for logged-in wrapper structure. Keep `WC_Shortcode_My_Account::output()` / shortcode as the source of truth for **endpoint content**—avoid reimplementing WC forms.
- **Hooks:** Use `woocommerce_account_navigation`, `woocommerce_account_content`, `woocommerce_account_dashboard`, and/or wrapper markup filters **only** if necessary; prefer **template overrides in the theme** under `woocommerce/myaccount/` **only** if the team already uses that pattern—otherwise keep logic in `src/page` components. **Do not** fork entire `form-edit-account.php` unless required.
- **CSS:** Extend `src/page/styles/*.scss` (e.g. partial for dashboard). Scope under `.page-account-block--logged-in` / `.etheme-contrive-account`. **No Bootstrap.** Tailwind utilities in markup allowed per existing block. Honor `prefers-reduced-motion`.
- **JS:** Add only if mobile nav or tabs need it; if so, `src/page/index/view.js` or a small imported module; progressive enhancement (works without JS for core navigation).
- **Editor:** Logged-in layout is not previewable in editor; keep **placeholder** honest (“Customer dashboard appears on the front when logged in”).
- **Escaping:** `esc_html`, `esc_attr`, `esc_url` everywhere.
- **Build:** Do not run `npm run build` unless the user asks; assume `npm run start:watch`.

**Out of scope (unless trivially included):** wishlists, third-party WC extensions UI, custom order tracking maps, rewriting WC emails.

**Acceptance criteria:**

1. Logged-in: sidebar + content layout matches breakpoint rules; default WC menu order and **custom items added by filters** still appear.
2. Every endpoint still loads and submits (orders list, address forms, account edit, logout).
3. Guest flows (login/register/lost password) **unchanged** visually and functionally vs Spec 15.
4. No PHP notices on typical account pages; no unescaped output.
5. Lighthouse/a11y: no obvious violations (headings not duplicated blindly—if hiding WC’s redundant `h2` for guests only was in Spec 15, do not break logged-in headings).

### Dudas resueltas por defecto en el prompt

- **Idioma UI:** Cadenas con `__()` / dominio `etheme` o textos de Woo existentes.
- **Descargas:** Si el cliente no tiene productos descargables, el ítem puede quedar vacío según WC; no forzar ocultar salvo filtro estándar y consentimiento del negocio (documentar en commit si se filtra).
