# Mails transaccionales (`woocommerce/emails/` + formulario de contacto)

## Alcance

Set completo de mails transaccionales del sitio: personalización visual coherente con la tienda, envío confiable vía SMTP externo, español AR con tuteo.

**Excepción explícita a [no-wc-overrides](.claude/rules/no-wc-overrides.md)**: esta spec habilita la carpeta `woocommerce/emails/` **únicamente para templates de email**. La prohibición sigue vigente para todos los demás templates del plugin (shop, cart, checkout, single-product, etc.). Dejar el motivo comentado en el README de la carpeta (`woocommerce/emails/README.md`) para que quede explícito.

### Mails cubiertos

**Al cliente (2):**

| # | Mail | Trigger WC | Template a overridear |
|---|------|-----------|-----------------------|
| C1 | Pago recibido / pedido en preparación | `woocommerce_order_status_processing_notification` | `customer-processing-order.php` |
| C2 | Pedido entregado | `woocommerce_order_status_completed_notification` | `customer-completed-order.php` |

**Al admin (3):**

| # | Mail | Trigger | Template |
|---|------|---------|----------|
| A1 | Nuevo pedido | `woocommerce_order_status_pending_to_processing_notification` (y `on-hold_to_processing`) | `admin-new-order.php` |
| A2 | Pedido fallido | `woocommerce_order_status_failed_notification` | `admin-failed-order.php` |
| A3 | Nuevo mensaje de formulario de contacto | Envío manual desde handler AJAX de contacto | `emails/admin-contact-message.php` (no es un mail WC, se renderiza con el helper compartido) |

**Mails WC que quedan como están (usan header/footer custom pero sin override de contenido):**

- `customer-on-hold-order`, `customer-refunded-order`, `customer-note`, `customer-reset-password`, `customer-new-account`, `admin-cancelled-order`. Heredan el `email-header.php`, `email-footer.php` y `email-styles.php` customizados por el tema, pero su cuerpo es el default de WC. Si más adelante se quieren personalizar, se agregan al set overrideado.

### Fuera de alcance (explícito, para no inventar)

- Carrito abandonado, producto back-in-stock, review request, newsletter — no se implementan en esta spec.
- Tracking de envío: el envío es manual, no hay integración con correo ni tracking number. El mail C2 (pedido entregado) se dispara cuando el admin marca el pedido como "Completado" a mano desde el backoffice.

## Tipo de implementación (obligatorio)

### 1. Estructura de archivos

```
woocommerce/emails/
├── README.md                          ← justifica la excepción a no-wc-overrides
├── email-header.php                   ← override del header global (logo + barra color)
├── email-footer.php                   ← override del footer global (datos fiscales, redes, legal)
├── email-styles.php                   ← override de los estilos inline (fuente, colores, spacing)
├── email-order-details.php            ← override del bloque de detalle de orden (tabla de ítems)
├── customer-processing-order.php      ← mail C1
├── customer-completed-order.php       ← mail C2
├── admin-new-order.php                ← mail A1
├── admin-failed-order.php             ← mail A2
└── plain/                             ← versiones texto plano (fallback obligatorio)
    ├── customer-processing-order.php
    ├── customer-completed-order.php
    ├── admin-new-order.php
    └── admin-failed-order.php

src/core/emails/
├── helpers.php                        ← etheme_send_contact_notification(), etheme_render_email()
├── contact-message-email.php          ← template del mail A3 (HTML)
└── contact-message-email-plain.php    ← versión texto plano del mail A3
```

### 2. Hooks y filtros a usar

Todo en [functions.php](functions.php) o en un archivo nuevo `src/core/emails/includes/email-hooks.php` incluido desde `functions.php`:

- **`woocommerce_email_styles`** — filtro que permite modificar los estilos inline del mail sin tocar el template. Útil si querés sobreescribir solo colores/fuente sin reemplazar `email-styles.php`.
- **`woocommerce_email_from_name`** y **`woocommerce_email_from_address`** — usar `etheme_get_core_config()` para leer nombre y mail del sitio.
- **`woocommerce_email_subject_*`** — filtros por mail (ej. `woocommerce_email_subject_customer_processing_order`) para customizar los asuntos en AR, con emoji si querés (opcional, ver sección diseño).
- **`woocommerce_email_heading_*`** — mismo patrón para el heading interno del mail.
- **Enviar el mail A3**: dentro de [src/contact/includes/](src/contact/includes/) (handler AJAX del formulario de contacto, existente), después de validar y sanitizar, llamar `etheme_send_contact_notification( $nombre, $email, $mensaje, $telefono )` que renderiza el template y despacha con `wp_mail()` + header `Content-Type: text/html`.

### 3. Clase custom de `WC_Email` para contacto (opcional, recomendado)

Para que el mail A3 aparezca en **WooCommerce → Ajustes → Emails** (y el admin pueda editar destinatario/asunto desde UI), crear una clase `Etheme_Contact_Message_Email extends WC_Email` y registrarla con el filtro `woocommerce_email_classes`. El handler AJAX entonces dispara la notificación con `WC()->mailer()->get_emails()['Etheme_Contact_Message_Email']->trigger( $data )`.

Si no se quiere acoplar el form de contacto a WC, dejarlo como `wp_mail()` plano — más simple, pero el admin no puede editar desde la UI de WC. **Recomendado: clase WC_Email**, para que el sistema de mails quede unificado y aproveche el header/footer/estilos custom automáticamente.

### 4. SMTP externo (prerrequisito de infra)

Los `wp_mail()` por PHP nativo tienen deliverability malísima (hosting compartido AR → muchos ISP marcan spam). Setup obligatorio antes de dar por cerrada la tarea:

**Recomendado: Fluent SMTP + Brevo (ambos gratis).**

1. Instalar plugin **FluentSMTP** (gratis, sin límite, mantenido).
2. Crear cuenta en **Brevo** (antes Sendinblue): 300 mails/día gratis para siempre. Verificar dominio con DKIM/SPF en el DNS.
3. En FluentSMTP, elegir Brevo como driver, pegar la API key.
4. Probar envío desde la UI del plugin.

**Alternativa: Resend.** 3.000 mails/mes gratis, 100/día. Mejor deliverability promedio que Brevo. Requiere verificar dominio también. FluentSMTP también lo soporta.

**No recomendado**: WP Mail SMTP Lite — funciona pero su versión free tiene menos drivers y el upsell es agresivo.

Dejar asentado en la spec que el setup SMTP es **responsabilidad de ops/cliente**, no del código del tema. El tema solo expone `etheme_get_core_config()` con el `from_name`/`from_address` correctos y los filtros de WC.

## Qué debe mostrar cada mail

### Estructura común (todos los mails)

1. **Header**: logo PNG centrado (ver nota SVG abajo) sobre barra de color coral `#fb704f` de 4px. Fondo del header blanco, 24px padding vertical.
2. **Body**: fondo `#f7f7f7`, card blanca centrada (ancho máx 600px, padding 32px), tipografía `Jost, Arial, sans-serif` (fallback obligatorio — muchos clientes de mail no cargan web fonts), tamaño base 16px, color `#333`.
3. **Footer**: tres bloques verticales, fondo gris claro:
   - Datos del negocio (nombre, CUIT si aplica, dirección, teléfono, mail) — desde `etheme_get_core_config()`.
   - Redes sociales como texto con links (no iconos SVG, por compatibilidad) — desde `core.json`.
   - Línea legal mínima: "Recibiste este mail porque realizaste una compra en [sitio]. No responder a este mail." (o similar para A3).

### Nota crítica sobre el logo

**SVG no renderiza en Gmail, Outlook (todas las versiones) ni Apple Mail.** Aproximadamente 80% de los clientes no lo muestran. **Prerrequisito**: exportar una versión PNG del logo:

- Archivo: `assets/images/logo-email.png`
- Tamaño: ~96px de alto, ancho proporcional
- Fondo transparente
- Peso: &lt;50 KB

El header del mail usa `<img src="{site_url}/wp-content/themes/bakka/assets/images/logo-email.png">` con `width`/`height` explícitos (requerido por Outlook).

Si al cerrar la tarea el PNG no existe, dejar `etheme_get_email_logo_url()` devolviendo un fallback al SVG + comentario `// TODO: reemplazar por PNG cuando esté disponible`, pero marcarlo en el criterio de aceptación como pendiente.

### C1 — Pago recibido / pedido en preparación

- **Asunto**: `Recibimos tu pedido #{order_number} — ya estamos preparándolo`
- **Heading del mail**: `¡Gracias por tu compra, {nombre}!`
- **Cuerpo**:
  - Párrafo 1: "Recibimos tu pedido y el pago está confirmado. Ya lo estamos preparando para enviártelo."
  - Párrafo 2: tiempo estimado de preparación (texto hardcodeado o leído de `core.json` → agregar campo `shipping.preparation_time` si no existe, ej. "3 a 5 días hábiles").
  - Bloque `email-order-details.php` (ítems + totales).
  - Dirección de envío si aplica.
  - Párrafo de cierre con mail/tel de contacto para dudas (desde `core.json`).
- **CTA inline**: link "Ver mi pedido" a `$order->get_view_order_url()` (solo si el usuario tiene cuenta).

### C2 — Pedido entregado

- **Asunto**: `Tu pedido #{order_number} fue entregado`
- **Heading del mail**: `Tu pedido llegó, {nombre}`
- **Cuerpo**:
  - Párrafo 1: "Te confirmamos que tu pedido fue entregado. Esperamos que disfrutes tu compra."
  - Bloque `email-order-details.php` resumido (solo ítems, sin totales).
  - Párrafo 2: pedido de feedback informal por mail (sin link a sistema de reviews — no lo hay): "Si algo no salió como esperabas, escribinos a {email_contacto}. Nos importa."
- **Sin CTA explícito** — mail de cierre de ciclo, no acción.

### A1 — Nuevo pedido (admin)

- **Asunto**: `[{site_name}] Nuevo pedido #{order_number} — {total}`
- **Heading**: `Nuevo pedido recibido`
- **Cuerpo**:
  - Datos del cliente: nombre, email, teléfono.
  - Bloque `email-order-details.php` completo.
  - Direcciones de envío + facturación.
  - Nota del cliente si existe (`$order->get_customer_note()`).
  - Link al admin: "Ver pedido en el backoffice" → `admin_url( 'post.php?post=' . $order->get_id() . '&action=edit' )`.

### A2 — Pedido fallido (admin)

- **Asunto**: `[{site_name}] Pedido #{order_number} falló — acción requerida`
- **Heading**: `Pedido fallido`
- **Cuerpo**:
  - Mismo layout que A1 pero con un bloque de alerta arriba (fondo rojo claro) explicando el estado: "El pago no se completó. Revisá el pedido en el backoffice."
  - Link al admin igual que A1.

### A3 — Nuevo mensaje de formulario de contacto (admin)

- **Asunto**: `[{site_name}] Nuevo mensaje de contacto de {nombre}`
- **Heading**: `Nuevo mensaje desde el formulario de contacto`
- **Cuerpo**:
  - Tabla con: Nombre, Email, Teléfono (si lo envió), Fecha.
  - Bloque con el mensaje completo (formateado con `wpautop` para respetar saltos de línea, luego `wp_kses_post`).
  - Nota al pie: "Para responder, usá el email del cliente directamente."

## Datos y seguridad

- **Sanitización** en el handler AJAX de contacto (A3): todo input usa los sanitizers centralizados del tema ([src/core/security/sanitizers.js](src/core/security/sanitizers.js) en front, PHP `sanitize_text_field` / `sanitize_email` / `wp_kses_post` en backend).
- **Nonce**: el handler AJAX de contacto ya verifica nonce según [php-escape-and-nonce](.claude/rules/php-escape-and-nonce.md). Mantener.
- **Escape en los templates de mail**: todos los valores dinámicos con `esc_html`, `esc_attr`, `esc_url`. Contenido que viene formateado de WC (direcciones, items) con `wp_kses_post`. Nunca `echo` crudo.
- **Headers del mail**: `Content-Type: text/html; charset=UTF-8`, `From: {site_name} <{from_email}>`, `Reply-To: {reply_to_email}` (leer de `core.json` si existe un mail específico de respuesta; si no, usar el mismo `from`).
- **A3 Reply-To**: debe ser el mail del usuario que envió el formulario de contacto — permite que el admin responda directo con "Responder". Sanitizar antes de ponerlo en el header con `sanitize_email` + verificar `is_email`.
- **Rate limiting**: el handler AJAX del formulario de contacto ya debería tener alguna protección (ver [spec 16.contact.md](specs/16.contact.md)). Esta spec no lo reimplementa.
- **Sin `error_log` de tracing**, según [errors-only-logging](.claude/rules/errors-only-logging.md). Solo loggear si `wp_mail()` devuelve `false` en A3, o si la clase WC_Email lanza excepción.

## Diseño

### Principios

- **Tablas, no flexbox**: Outlook usa motor de Word, flex/grid no funcionan. Todo layout con `<table>` + `cellpadding`/`cellspacing`/`border="0"`.
- **Estilos inline**: casi todos los clientes de mail ignoran `<style>` en `<head>`. Usar atributos inline `style="..."` en cada elemento. Los estilos de `email-styles.php` existen para los pocos clientes que sí leen `<style>` (Apple Mail, Gmail web con algunas limitaciones).
- **Ancho máximo 600px**: estándar histórico de mail, garantiza que funcione en mobile y desktop.
- **Imágenes con `width`/`height` explícitos**: Outlook los necesita como atributos HTML, no solo CSS.
- **Sin JS**: todos los clientes lo strippean.
- **Sin SVG inline**: ya explicado.
- **Sin web fonts custom**: solo sans-serif sistémicas. Definir stack `Jost, 'Helvetica Neue', Helvetica, Arial, sans-serif` (Jost se carga en el sitio; en el mail cae a Arial en 90% de clientes).
- **Dark mode**: definir `meta name="color-scheme" content="light only"` en el `<head>` para evitar que Apple Mail invierta colores sin avisar. Gmail y Outlook tienen detección automática — aceptar que se van a ver un poco distintos.

### Tokens visuales

| Token | Valor |
|-------|-------|
| Primario / acento | `#fb704f` (coral, mismo del sitio) |
| Fondo body | `#f7f7f7` |
| Fondo card | `#ffffff` |
| Texto principal | `#333333` |
| Texto secundario | `#666666` |
| Borde tenue | `#e5e5e5` |
| Alerta (A2) | fondo `#fff1f0`, borde `#ff4d4f`, texto `#a8071a` |
| Success sutil (C1, C2) | opcional, franjita coral en el card header |
| Radio de bordes | `0` — bordes cuadrados, coherente con la tienda |

### Tipografía

- Heading principal del mail: 24px, peso 600, color `#333`.
- Subheading / nombre del cliente: 18px, peso 500.
- Body: 16px, line-height 1.6.
- Legal / pie: 13px, color `#666`.

### Ejemplos de asuntos (español AR con tuteo)

- C1: `Recibimos tu pedido #1042 — ya estamos preparándolo`
- C2: `Tu pedido #1042 fue entregado`
- A1: `[Bakka] Nuevo pedido #1042 — $45.000`
- A2: `[Bakka] Pedido #1042 falló — acción requerida`
- A3: `[Bakka] Nuevo mensaje de contacto de María Pérez`

Sin emojis por defecto (riesgo de verse mal en clientes viejos). Si el cliente los quiere, agregarlos al final del asunto donde impactan menos el preview.

## Contexto técnico del repo (no inventar rutas)

- **Patrón existente a imitar**: no hay otros overrides de mail en este tema — es el primer caso. Seguir la estructura estándar de WooCommerce (carpeta `woocommerce/emails/` en la raíz del tema, junto a `functions.php`, `templates/`, `parts/`).
- **Config del sitio**: [src/core/config/config.json](src/core/config/config.json) — usar para `from_name`, `from_address`, `reply_to`, redes sociales del footer, datos del negocio, mail de admin donde llegan A1/A2/A3. Si falta algún campo, agregarlo siguiendo el patrón existente y documentar en [docs/config-json.md](docs/config-json.md).
- **Handler AJAX del formulario de contacto**: ubicado en `src/contact/includes/` según [spec 16.contact.md](specs/16.contact.md). Agregar ahí la llamada a `etheme_send_contact_notification()` o al trigger de la clase `Etheme_Contact_Message_Email`.
- **Logo**: PNG en `assets/images/logo-email.png` (prerrequisito, responsabilidad del cliente/diseño). El helper `etheme_get_email_logo_url()` vive en `src/core/emails/helpers.php`.
- **No crear nuevos endpoints AJAX** — el de contacto ya existe.
- **No modificar** el sistema de envío nativo de WC (no tocar `WC_Emails::send()` ni `WC_Mailer`). Solo overrides de templates + filtros declarativos.
- **Registro de la clase custom**: en `functions.php` o en `src/core/emails/includes/email-hooks.php`, hookear `woocommerce_email_classes` y agregar la clase `Etheme_Contact_Message_Email` al array.

## Criterios de aceptación

1. **Carpeta `woocommerce/emails/`** creada con los 9 templates listados (4 mails cliente/admin + 4 en `plain/` + `README.md` que justifica la excepción).
2. **Los 5 mails se disparan correctamente**:
   - C1 al pasar pedido a "Procesando" (pago recibido).
   - C2 al pasar pedido a "Completado" (admin lo marca manual).
   - A1 al crearse un pedido nuevo que entra a procesando.
   - A2 cuando un pedido falla (pago rechazado).
   - A3 al enviar el formulario de contacto con datos válidos.
3. **Look visual** coherente con el sitio: logo (PNG) en header sobre barra coral, tipografía sans-serif, fondo `#f7f7f7`, card blanca, bordes cuadrados, CTAs coral.
4. **Compatibilidad** verificada en los 3 clientes que cubren el 90% del mercado AR:
   - Gmail web + app Android/iOS.
   - Outlook web + desktop Windows.
   - Apple Mail macOS + iOS.
   - Usar [Litmus](https://litmus.com) o [Email on Acid](https://www.emailonacid.com) en trial, o enviar a cuentas reales de prueba. Mínimo: layout no se rompe, logo se ve, colores no se invierten en dark mode.
5. **Asuntos en español AR con tuteo**, formato consistente (`[{site}]` para admin, sin brackets para cliente).
6. **Versiones plain text** generadas para los 4 mails de WC overrideados (no opcional — requerido para deliverability).
7. **Reply-To del A3** es el email del que envió el formulario, no el del sitio.
8. **SMTP configurado**: FluentSMTP instalado, conectado a Brevo (o Resend), dominio verificado con SPF+DKIM, prueba de envío OK. Dejarlo documentado en el handoff aunque no sea código del tema.
9. **PNG del logo** presente en `assets/images/logo-email.png`, o en su defecto TODO claramente marcado en `etheme_get_email_logo_url()` y flag en el handoff.
10. **Sin regresiones** en los otros mails nativos de WC (on-hold, refunded, customer-note, reset-password, new-account, admin-cancelled) — siguen funcionando con el header/footer/estilos custom aplicados.
11. **Sin `console.log` ni `error_log` de tracing**. Solo `error_log` si `wp_mail()` devuelve false en A3.
12. **Escapado**: todo valor dinámico en templates PHP de mail usa `esc_html` / `esc_attr` / `esc_url` / `wp_kses_post` según el caso.
13. **Setup documentado** en [docs/EMAILS.md](docs/EMAILS.md): cómo configurar SMTP, cómo testear cada mail desde el admin WC (WooCommerce → Estado → Herramientas → "Enviar email de prueba"), cómo regenerar el PNG del logo si cambia la marca.
