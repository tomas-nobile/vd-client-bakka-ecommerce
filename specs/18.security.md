## Alcance técnico

### 1) Centralización en `src/core` (obligatorio)

Crear módulos reutilizables en `src/core` para evitar lógica duplicada por página:

- `src/core/security/validators.js`
  - validadores puros: required, email, phone, numeric, length, regex, postcode AR (4 dígitos), personName (rechaza dígitos), etc.
- `src/core/security/sanitizers.js`
  - sanitizado de strings para cliente: trim, normalización whitespace, limpieza de caracteres de control.
- `src/core/security/request-guard.js`
  - helpers para requests: timeout, abort controller, retry acotado (solo para errores de red), payload guard, bloqueo doble submit, normalización de errores.
- `src/core/security/messages.js`
  - catálogo central de mensajes de validación/estado (español), reusable por cupón/checkout/contacto.
  - incluye `FIELD_LABELS`: mapa de nombre de campo → etiqueta legible en español, para usar en el banner de errores del checkout.
- `src/core/security/ui-feedback.js`
  - utilidades para UX de errores: inline field error, estado `aria-invalid`, resumen global opcional, estado loading/success/error.

Si ya existen utilidades equivalentes en el proyecto, reutilizar y extender en lugar de duplicar.

### 2) Cupón (`src/page-cart`)

Aplicar seguridad en flujo de cupón:

- Validar y sanitizar `coupon_code` antes del submit.
- Reglas mínimas:
  - requerido
  - largo mínimo/máximo razonable
  - permitir solo set esperado de caracteres (evitar payloads obvios de inyección)
- Bloquear doble submit mientras request está en curso.
- No usar `innerHTML` para mensajes no confiables del usuario.
- Mensajes amigables con diseño consistente:
  - campo vacío
  - formato inválido
  - error de red
  - éxito de aplicación/remoción

### 3) Checkout 2 pasos (`src/page-checkout`)

Extender y unificar validación existente sin romper WooCommerce:

- Reusar `checkout-validation.js` y `checkout-region-guard.js` donde corresponda.
- Normalizar validaciones con helpers de `src/core/security`.
- Reforzar en paso 1:
  - email válido
  - teléfono numérico (área y número separados)
  - código postal 4 dígitos
  - campos requeridos
  - restricción geográfica vigente (CABA + GBA)
  - nombre y apellido: solo letras, espacios, guiones, apóstrofes y puntos — rechazar dígitos y símbolos
- Evitar bypass de botón "Continuar al pago" por estados inconsistentes.
- Mantener accesibilidad:
  - `aria-invalid` en cada campo inválido
  - `aria-live` para mensajes inline y banner global
  - foco al primer campo inválido al intentar avanzar
- Mostrar feedback en dos niveles:
  - **Inline por campo:** error específico debajo del input (`.etheme-field-error`).
  - **Banner global encima del botón "Continuar al pago":** lista de errores específicos con etiqueta de campo legible. Por ejemplo: `Email: Ingresá un email válido.` / `Apellido: Ingresá solo letras en este campo.`
    - El banner NO muestra un mensaje genérico sino la lista real de problemas pendientes.
    - Se actualiza en tiempo real a medida que el usuario corrige cada campo.
    - Solo aparece en campos que el usuario ya tocó (no anuncia errores en campos vacíos intactos).
    - No se muestra si el bloqueo activo es por restricción geográfica (zona no permitida).

### 4) Contacto (`src/contact`)

Mantener post directo al endpoint AWS pero endurecer cliente:

- Validar/sanitizar:
  - nombre
  - teléfono
  - email
  - mensaje
- Agregar anti-abuso liviano en cliente:
  - honeypot oculto (si se completa, bloquear envío silenciosamente)
  - mínimo tiempo humano de envío (timestamp gate, por ejemplo >= 2-3 segundos)
  - cooldown corto entre envíos para evitar flood por usuario/sesión
- Request hardening:
  - timeout de fetch
  - manejo consistente de errores de red/API
  - no exponer detalles sensibles en mensajes UI
- UX:
  - errores inline por campo + mensaje global de estado (`loading/success/error`)
  - textos amigables y claros
  - diseño consistente con clases actuales (`.etheme-field-error`, `.etheme-form-notice`) mejoradas si hace falta

## Requisitos de seguridad a contemplar

Aunque esta tarea se enfoque en frontend, deja el código preparado para convivir con defensas backend. Considerar y reflejar en decisiones:

1. XSS: nunca inyectar input no confiable en HTML sin escape.
2. HTML injection: usar `textContent` cuando el contenido venga de usuario.
3. CSRF: si hay endpoints WordPress/AJAX en estos flujos, incluir nonce y verificación correspondiente.
4. AJAX abuse: añadir controles de frecuencia del lado cliente y estructura lista para rate limit server-side.
5. Parameter pollution: validar tipos esperados y normalizar entrada.
6. REST/API abuse: no asumir que frontend es defensa suficiente.
7. SQL Injection: aunque no se resuelve desde JS, exigir que cualquier query dinámica en PHP use `$wpdb->prepare()` y que no se concatene input en SQL (incluye helpers/filtros WooCommerce y búsquedas custom).
8. Open redirect / command injection: no aplica directo en estos scripts, pero evitar patrones que lo faciliten.
9. Business Logic Attacks (WooCommerce): impedir combinaciones inválidas de flujo (bypass de stepper, uso de estados inconsistentes, cupones en estado inválido de carrito, race conditions de submit múltiple).
10. Cache Poisoning / fuga de datos por caché: asegurar que cart/checkout/my-account no sean cacheables por edge/page cache y que ninguna respuesta dinámica de usuario se comparta entre sesiones.

Además, proponer mejoras adicionales de alto impacto (WAF/rate limit server-side/CORS estricto/observabilidad) en notas finales.

### Implementación mínima obligatoria para los 3 riesgos agregados

1. **SQLi (theme/backend contract):**
   - Auditar en esta tarea los puntos tocados en PHP relacionados a entradas de usuario.
   - Si aparece SQL manual, reemplazar por `$wpdb->prepare()`.
   - Documentar en la respuesta final qué puntos se revisaron y si hubo cambios.
2. **Business Logic WooCommerce:**
   - Blindar el flujo de checkout para que no avance si el estado de validación es inválido.
   - Bloquear dobles envíos en contacto/cupón/checkout para reducir race conditions.
   - No confiar en estado visual del botón: recalcular precondiciones antes de cada acción crítica.
3. **Cache Poisoning:**
   - Verificar que páginas de WooCommerce sensibles no se marquen como cacheables en esta capa del theme.
   - Si aplica, añadir headers defensivos o compatibilidad con plugins de caché mediante hooks/filtros recomendados por WooCommerce.
   - Incluir recomendación explícita para CDN/WAF en riesgos residuales.
